# Research: CloudNativePG PostgreSQL Cluster

**Feature**: 012-cloudnative-pg  
**Date**: 2026-01-07

## 1. CloudNativePG Operator Installation

### Decision
Use official CloudNativePG Helm chart from `https://cloudnative-pg.github.io/charts` deployed via Flux HelmRelease.

### Rationale
- Official chart maintained by CloudNativePG project
- CNCF sandbox project with active development
- Helm chart supports GitOps workflows with Flux
- Lightweight operator suitable for homelab (single controller pod)

### Alternatives Considered
| Alternative | Rejected Because |
|-------------|------------------|
| Bitnami CloudNativePG chart | Additional abstraction layer, less direct control |
| Zalando Postgres Operator | More complex, designed for larger deployments |
| CrunchyData PGO | Enterprise-focused, heavier resource requirements |
| Raw PostgreSQL StatefulSet | No operator benefits (failover, backup management) |

## 2. Namespace Strategy

### Decision
- Operator: `cnpg-system` namespace
- PostgreSQL cluster: `postgres` namespace

### Rationale
- Follows CloudNativePG documentation conventions
- Separates operator lifecycle from database lifecycle
- Allows different RBAC policies per namespace
- Consistent with existing patterns (longhorn-system for operator, separate namespace for resources)

### Alternatives Considered
| Alternative | Rejected Because |
|-------------|------------------|
| Single namespace for both | Operator upgrades could disrupt database |
| Operator in flux-system | Violates separation of concerns |

## 3. Backup Strategy

### Decision
Use CloudNativePG native backup to S3-compatible storage (Cloudflare R2) with:
- `barmanObjectStore` configuration in Cluster CRD
- `ScheduledBackup` CRD for daily backups
- 7-day retention policy

### Rationale
- CloudNativePG has built-in Barman integration for S3 backups
- R2 credentials already exist for Longhorn backups (can create separate or reuse pattern)
- Native backup includes WAL archiving for point-in-time recovery
- No additional backup tools required

### Alternatives Considered
| Alternative | Rejected Because |
|-------------|------------------|
| Longhorn volume snapshots only | No point-in-time recovery, larger backup size |
| External backup tool (pgBackRest) | Additional complexity, CloudNativePG has built-in |
| Manual pg_dump scripts | No automation, no WAL archiving |

## 4. Storage Configuration

### Decision
Use Longhorn StorageClass with 10GB PVC for PostgreSQL data.

### Rationale
- Longhorn already deployed and operational
- 10GB sufficient for homelab multi-tenant use
- Longhorn provides additional snapshot capability
- Single replica count matches homelab single-node setup

### Alternatives Considered
| Alternative | Rejected Because |
|-------------|------------------|
| Local path provisioner | No redundancy, no snapshots |
| hostPath volume | Not portable, single-node only |
| Larger storage (20GB+) | Over-provisioning for current needs |

## 5. PostgreSQL Version

### Decision
PostgreSQL 16 using `ghcr.io/cloudnative-pg/postgresql:16` image.

### Rationale
- PostgreSQL 16 is current stable release
- Good balance of features and stability
- CloudNativePG provides optimized container images
- Supports all required features (SCRAM auth, logical replication)

### Alternatives Considered
| Alternative | Rejected Because |
|-------------|------------------|
| PostgreSQL 15 | Older, fewer performance improvements |
| PostgreSQL 17 | Very new, less tested |
| Custom PostgreSQL image | Maintenance burden, CloudNativePG images are optimized |

## 6. Credentials Management

### Decision
- Superuser credentials: Auto-generated by CloudNativePG, stored in cluster-managed Secret
- R2 backup credentials: SOPS-encrypted Secret following existing pattern

### Rationale
- CloudNativePG generates secure random passwords automatically
- Application credentials created manually via SQL (documented procedure)
- R2 credentials follow existing `secret-*.sops.yaml` pattern
- Secrets accessible via cluster DNS after decryption

### Alternatives Considered
| Alternative | Rejected Because |
|-------------|------------------|
| Pre-defined superuser password | Less secure, manual password management |
| External secrets operator | Additional complexity for homelab |

## 7. Helm Chart Configuration

### Decision
Minimal HelmRelease values - use chart defaults for operator installation.

### Rationale
- CloudNativePG operator defaults are suitable for single-node deployment
- No need for HA operator configuration
- Reduces maintenance burden
- Chart version pinned for stability (renovate will update)

### Key Values
```yaml
# helmrelease.yaml spec.values
# Using defaults - operator runs single replica controller
```

## 8. Cluster CRD Configuration

### Decision
Single-instance cluster with Longhorn storage and R2 backup.

### Key Configuration
```yaml
apiVersion: postgresql.cnpg.io/v1
kind: Cluster
metadata:
  name: postgres-cluster
  namespace: postgres
spec:
  instances: 1
  imageName: ghcr.io/cloudnative-pg/postgresql:16
  storage:
    size: 10Gi
    storageClass: longhorn
  backup:
    retentionPolicy: "7d"
    barmanObjectStore:
      destinationPath: "s3://homelab-postgres-backups/cnpg/"
      endpointURL: "https://<account-id>.r2.cloudflarestorage.com"
      s3Credentials:
        accessKeyId:
          name: postgres-r2-credentials
          key: ACCESS_KEY_ID
        secretAccessKey:
          name: postgres-r2-credentials
          key: ACCESS_SECRET_KEY
```

## 9. Service Endpoints

### Decision
Use CloudNativePG auto-created Services:
- `postgres-cluster-rw`: Read-write endpoint (primary)
- `postgres-cluster-r`: Read-only endpoint (replicas, if any)
- `postgres-cluster-ro`: Read-only endpoint (alias)

### Rationale
- CloudNativePG automatically creates these Services
- DNS names follow predictable pattern
- Applications connect via `postgres-cluster-rw.postgres.svc.cluster.local`

## 10. Monitoring Integration

### Decision
Defer to existing Grafana Alloy setup - CloudNativePG exposes Prometheus metrics by default.

### Rationale
- CloudNativePG operator exposes `/metrics` endpoint
- PostgreSQL instances expose metrics via built-in exporter
- Grafana Alloy can scrape these endpoints
- Dashboard configuration is separate concern (future enhancement)
