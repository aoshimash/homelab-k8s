# Feature Specification: K8s Lint & Security Checks

**Feature Branch**: `007-k8s-lint-security`  
**Created**: 2026-01-03  
**Status**: Draft

## User Scenarios & Testing *(mandatory)*

### User Story 1 - ローカルでK8sマニフェストのリントチェックを実行する (Priority: P1)

開発者として、K8sマニフェストを変更した後、ローカル環境でリントチェックを実行して、構文エラーやベストプラクティス違反を検出したい。

**Why this priority**: マニフェストの品質を開発初期段階で確認できることで、CIでの失敗を減らし、開発効率を向上させる。

**Independent Test**: `task lint` コマンドを実行し、リントエラーがある場合はエラー内容が表示され、エラーがない場合は成功メッセージが表示されることを確認する。

**Acceptance Scenarios**:

1. **Given** K8sマニフェストに構文エラーがある, **When** リントチェックを実行する, **Then** エラー箇所とエラー内容が明確に表示される
2. **Given** K8sマニフェストに構文エラーがない, **When** リントチェックを実行する, **Then** チェック成功が表示される
3. **Given** K8sマニフェストがKubernetesベストプラクティスに違反している, **When** リントチェックを実行する, **Then** 違反内容と推奨される修正方法が表示される

---

### User Story 2 - ローカルでK8sマニフェストのセキュリティチェックを実行する (Priority: P1)

開発者として、K8sマニフェストを変更した後、セキュリティ脆弱性や設定ミスを検出したい。

**Why this priority**: セキュリティ問題を本番環境にデプロイする前に検出することで、セキュリティリスクを最小限に抑える。

**Independent Test**: `task security` コマンドを実行し、セキュリティ問題がある場合は警告が表示され、問題がない場合は成功メッセージが表示されることを確認する。

**Acceptance Scenarios**:

1. **Given** K8sマニフェストにセキュリティ脆弱性がある（例：privilegedコンテナ、不適切なsecurityContext）, **When** セキュリティチェックを実行する, **Then** 脆弱性の詳細と深刻度が表示される
2. **Given** K8sマニフェストにセキュリティ脆弱性がない, **When** セキュリティチェックを実行する, **Then** チェック成功が表示される
3. **Given** K8sマニフェストが既知のセキュリティ設定ミスを含む, **When** セキュリティチェックを実行する, **Then** 設定ミスの詳細と修正方法が表示される

---

### User Story 3 - ローカルで全チェックを一括実行する (Priority: P2)

開発者として、コミット前に全てのチェック（リント、セキュリティ）を一括で実行したい。

**Why this priority**: 複数のチェックを個別に実行する手間を省き、コミット前の品質確認を効率化する。

**Independent Test**: `task check` コマンドを実行し、全チェックが順番に実行され、総合結果が表示されることを確認する。

**Acceptance Scenarios**:

1. **Given** 複数のチェックが設定されている, **When** 全チェック実行コマンドを実行する, **Then** 全てのチェックが順番に実行され、各チェックの結果が表示される
2. **Given** いずれかのチェックで問題が検出された, **When** 全チェックを実行する, **Then** 問題があったチェックの詳細と全体の失敗ステータスが表示される
3. **Given** 全てのチェックで問題がない, **When** 全チェックを実行する, **Then** 全チェック成功メッセージが表示される

---

### User Story 4 - CIでPull Request時にリントとセキュリティチェックを自動実行する (Priority: P1)

リポジトリ管理者として、Pull Requestが作成・更新された際に、自動的にリントとセキュリティチェックを実行し、問題があればマージをブロックしたい。

**Why this priority**: 不適切なマニフェストがメインブランチにマージされることを防ぎ、リポジトリの品質を維持する。

**Independent Test**: Pull Requestを作成し、GitHub Actionsが自動的にチェックを実行し、結果がPRに表示されることを確認する。

**Acceptance Scenarios**:

1. **Given** Pull Requestが作成された, **When** CIパイプラインが実行される, **Then** リントチェックとセキュリティチェックが実行される
2. **Given** チェックで問題が検出された, **When** CIパイプラインが完了する, **Then** チェックステータスが失敗となり、PRに詳細なエラーレポートが表示され、マージがブロックされる
3. **Given** 全てのチェックが成功した, **When** CIパイプラインが完了する, **Then** チェックステータスが成功となり、マージが可能になる

---

### User Story 5 - ローカルとCIで同一のチェックを実行する (Priority: P1)

開発者として、ローカルで実行したチェックとCIで実行されるチェックが同一であることを保証したい。

**Why this priority**: ローカルとCIで異なるチェック結果になることを防ぎ、"ローカルでは通ったのにCIで失敗"という問題を解消する。

**Independent Test**: ローカルで実行したチェックコマンドとCIで実行されるコマンドが同一であり、同じ結果を返すことを確認する。

**Acceptance Scenarios**:

1. **Given** 同一のK8sマニフェスト, **When** ローカルとCIで同じチェックを実行する, **Then** 同じ結果が得られる
2. **Given** チェックツールのバージョン管理が設定されている, **When** ローカルとCIでチェックを実行する, **Then** 同一バージョンのツールが使用される

---

### Edge Cases

- **空のマニフェストディレクトリ**: k8sディレクトリにマニフェストファイルが存在しない場合、エラーではなく適切なメッセージを表示する
- **無効なYAML構文**: YAMLとしてパースできないファイルがある場合、明確なエラーメッセージを表示する
- **大量のマニフェストファイル**: 多数のファイルがある場合でも、適切な時間内にチェックが完了する
- **チェックツールが未インストール**: 必要なツール（trivy等）がインストールされていない場合、インストール方法を案内する

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムはK8sマニフェストファイル（.yaml, .yml）に対してkubeconformを使用したリントチェック（スキーマ検証）を実行できなければならない
- **FR-002**: システムはK8sマニフェストファイルに対してTrivyを使用したセキュリティチェックを実行できなければならない
- **FR-003**: システムはTaskfileを使用して複数のチェックタスクを定義し、一括実行できなければならない
- **FR-004**: システムはGitHub Actionsを通じてPull Request時に自動的にチェックを実行し、失敗時はRequired status checkとしてマージをブロックしなければならない
- **FR-005**: ローカルとCI環境で同一のチェックロジックを使用しなければならない
- **FR-006**: チェック結果は明確なフォーマットで出力され、問題箇所と修正方法が分かりやすく表示されなければならない
- **FR-007**: チェックで問題が検出された場合、非ゼロの終了コードを返さなければならない
- **FR-008**: システムはk8sディレクトリ配下の全てのマニフェストファイルを対象にチェックを実行しなければならない

### Key Entities

- **Taskfile**: ローカルでのチェックタスクを定義するファイル。lint、security、check（一括実行）タスクを含む
- **GitHub Actions Workflow**: CIでのチェックを定義するワークフローファイル。ローカルと同一のチェックを実行する
- **K8sマニフェスト**: チェック対象となるKubernetesリソース定義ファイル

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 開発者がローカルでリントチェックを実行し、結果を確認するまで30秒以内で完了する
- **SC-002**: 開発者がローカルでセキュリティチェックを実行し、結果を確認するまで60秒以内で完了する
- **SC-003**: Pull Request作成時にCIチェックが自動実行され、5分以内に結果が表示される
- **SC-004**: ローカルとCIで同一のマニフェストに対するチェック結果が100%一致する
- **SC-005**: チェックで検出された全ての問題に対して、修正に必要な情報が提供される

## Assumptions

- 開発者のローカル環境にTaskがインストールされている（またはインストール可能である）
- 開発者のローカル環境にkubeconformがインストールされている（またはインストール可能である）
- 開発者のローカル環境にTrivyがインストールされている（またはインストール可能である）
- GitHubリポジトリでGitHub Actionsが有効化されている
- チェック対象のK8sマニフェストは `k8s/` ディレクトリ配下に配置されている
- リントチェックには一般的なKubernetesベストプラクティスを適用する
