# Incident: Talos 1.13 + Cilium 1.19 + Tailscale extension multi-failure

- **Date**: 2026-05-04 (Asia/Tokyo)
- **Cluster**: homelab-k8s, single-node Talos at `192.168.0.10`
- **Duration**: ~2.5 hours of active outage; further ~1 hour of cosmetic diagnosis
- **User-visible symptom**: paperless-ngx, Tailscale ingress-proxies, and all four Flux controllers in CrashLoopBackOff. Cluster appeared "broken" after a routine Talos OS + Cilium upgrade window.
- **Final outcome**: cluster fully recovered. One harmless cosmetic side effect (Cilium agent log noise) is left documented as accepted; root cause and constraints understood.

## TL;DR

After upgrading Talos OS from 1.12.0 to 1.13.0 and merging the Renovate PR that
bumped Cilium from 1.18.6 to 1.19.3 in the same window, the cluster fell into a
multi-component outage. Three distinct problems compounded:

1. **Cilium iptables masquerade broken** because the Cilium agent's iptables
   manager could not read the host's `nat` table on Talos 1.13 — the
   `ts-postrouting` chain written natively by `tailscaled` made the table
   unreadable through `iptables-nft`. Pod-to-external SNAT therefore never
   happened, breaking anything that needed outbound internet.
2. **Flux liveness probes blocked** because Flux's bundled NetworkPolicies
   (`allow-egress`, `allow-scraping`, `allow-webhooks`) selected every pod in
   `flux-system` with `from` clauses that match other pods/namespaces but never
   the host network. Cilium 1.19's stricter NetworkPolicy enforcement turned
   those policies from "lenient ignore" into "hard deny" for kubelet probes on
   ports 9440 / 9090.
3. **Flux GitOps reconciliation deadlocked** because (1) and (2) put all four
   Flux controllers in CrashLoopBackOff, so Flux could not apply the very
   Cilium upgrade that would have fixed (1).

The fix sequence was:
- Manual `helm upgrade cilium` to 1.19.3 with `bpf.masquerade=true` to break
  the deadlock.
- Delete the over-broad NetworkPolicies and remove them from
  `gotk-components.yaml` so Flux does not re-create them.
- Commit `bpf.masquerade=true` to the Cilium HelmRelease in git so Flux
  reconciliation matches the imperative state.

A follow-up attempt to silence the Cilium iptables log noise (which persists
even after the dataplane is restored) hit two upstream constraints and was
reverted; that part is captured below as a permanent finding.

## Cluster baseline at the start of the incident

- Talos OS: 1.12.0 with kernel from siderolabs/pkgs at the time.
- Kubernetes: 1.35.x.
- Cilium: 1.18.6, kubeProxyReplacement=true, masquerade via host iptables
  (default), routing-mode=tunnel (vxlan), MTU=1450.
- Tailscale: extension `siderolabs/tailscale@1.96.4` running on the node, plus
  the in-cluster Tailscale operator.
- Flux: v2.7.5, bootstrapped with the default `flux-system` NetworkPolicies.

## Symptoms (raw evidence)

### Cilium agent

```
iptables rules full reconciliation failed, will retry another one later
error="failed to remove old backup rules: unable to run 'iptables -t nat -S' iptables command:
exit status 1 stderr=\"iptables v1.8.8 (nf_tables): table 'nat' is incompatible, use 'nft' tool.\""
```

This line repeated every 10 seconds.

### paperless-ngx init container

```
Connecting to github.com (20.27.177.113:443)
wget: can't connect to remote host (20.27.177.113): Connection timed out
```

DNS resolved fine; TCP egress timed out — classic missing-SNAT signature.

### Tailscale ingress-proxies

```
bootstrapDNS("derp10.tailscale.com", "2001:19f0:8001:2d9:5400:2ff:feef:bbb1") for "log.tailscale.com" error:
  Get "https://derp10.tailscale.com/bootstrap-dns?q=log.tailscale.com": dial tcp [...]:443: connect: network is unreachable
```

Same egress-broken signature.

### Flux source-controller (representative — all four had the same pattern)

```
... starting server name="health probe" addr="[::]:9440"
... attempting to acquire leader lease flux-system/source-controller-leader-election...
(30 seconds elapse)
... Stopping and waiting for non leader election runnables
... Wait completed, proceeding to shutdown the manager
```

Exit code 0, `reason: Completed`, `Killing: Container manager failed liveness probe, will be restarted`.

The pod started cleanly, listened on `:9440`, then got SIGTERM exactly 30 seconds later — three liveness probe failures × 10 second period.

## Investigation: what we tried, in order

This section is sequenced so a future reader can replay it on the cluster.

### Step 1 — confirm the Cilium agent is healthy

```bash
kubectl -n kube-system get pods -l k8s-app=cilium
kubectl -n kube-system get cm cilium-config -o jsonpath='{.data.kube-proxy-replacement}'
kubectl -n kube-system get cm cilium-config -o jsonpath='{.data.enable-bpf-masquerade}'
```

Expected before the fix: agent Running but `enable-bpf-masquerade` is empty
(default false). The agent itself is healthy; the dataplane misconfig is the
problem.

### Step 2 — confirm masquerade is broken in iptables

The smoking gun in the agent log is the `iptables ... (nf_tables): table 'nat'
is incompatible, use 'nft' tool` line. Run:

```bash
kubectl -n kube-system logs ds/cilium --tail=200 | grep -i "iptables\|incompat"
```

If you see the line repeat at a 10-second cadence, this is the same incident.

### Step 3 — read the actual kernel netfilter state

This is where the diagnosis went from "Cilium error" to "Tailscale wrote
nft-native rules". Talos does not ship `nft` in any user-accessible place, and
`kubectl exec` into the Cilium pod is restricted on this cluster, so the
diagnostic uses a privileged debug Job:

```yaml
# /tmp/nft-inspector.yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: nft-inspector
  namespace: kube-system
spec:
  ttlSecondsAfterFinished: 300
  template:
    spec:
      hostNetwork: true
      hostPID: true
      restartPolicy: Never
      containers:
        - name: nft
          image: nicolaka/netshoot:latest
          command: ["/bin/sh", "-c"]
          args:
            - |
              echo "===== nft list tables ====="
              nft list tables
              echo ""
              echo "===== nft list ruleset ====="
              nft list ruleset
              echo ""
              echo "===== iptables-nft -t nat -S (the failing probe) ====="
              iptables-nft -t nat -S
          securityContext:
            privileged: true
            capabilities:
              add: ["NET_ADMIN", "NET_RAW", "SYS_ADMIN"]
```

Apply, wait, dump:

```bash
kubectl apply -f /tmp/nft-inspector.yaml
kubectl -n kube-system wait --for=condition=Complete job/nft-inspector --timeout=90s
kubectl -n kube-system logs job/nft-inspector
kubectl -n kube-system delete job nft-inspector
```

The relevant excerpt during the incident was:

```
table ip nat {
    chain POSTROUTING {
        type nat hook postrouting priority srcnat; policy accept;
        counter packets 86962 bytes 5281833 jump ts-postrouting
    }

    chain ts-postrouting {
        meta mark & 0x0000ff00 == 0x00000400 counter packets 0 bytes 0 masquerade
    }
    ...
}

# table ip filter (separate listing)
# Warning: table ip filter is managed by iptables-nft, do not touch!
table ip filter {
    ...
}
```

Two key observations:

1. The `ts-postrouting` chain is Tailscale's (the `ts-` prefix is its
   convention).
2. The `ip nat` table has **no** `# Warning: managed by iptables-nft, do not
   touch!` comment, while the sibling `ip filter` table does. That comment is
   added by `iptables-nft` whenever it writes a rule. Its absence on `ip nat`
   means those rules were written by raw `nft` (libnftnl), not by `iptables-nft`.

Running `iptables-nft -t nat -S` from the same inspector pod reproduces the
exact same error Cilium logs. So the binary is fine and the kernel state is
consistent — the table itself just contains a chain `iptables-nft` cannot
translate back to iptables form.

### Step 4 — confirm Tailscale is the writer

```bash
talosctl --talosconfig infra/talos/clusterconfig/talosconfig --nodes 192.168.0.10 \
  get extensionserviceconfigs tailscale -o yaml
```

In the configuration captured during the incident, the extension service had
only `TS_AUTHKEY` set — no `TS_DEBUG_FIREWALL_MODE`. With `tailscaled`
defaulting to `auto`, on a kernel with nf_tables support (Talos 1.13) it
selects `nftables` mode and writes its rules natively via libnftnl. That
matches exactly what the inspector pod showed.

### Step 5 — explain why it broke at the Talos 1.13 boundary specifically

Tailscale was running its `auto` netfilter mode on Talos 1.12 too, and Cilium
1.18.6 was already periodically probing `iptables -t nat -S`. So in principle
this could have been broken on 1.12 too. The likely trigger of the Talos 1.13
upgrade is one or more of:

- The kernel jump to 6.18.24 changed how nf_tables represents certain chain
  expressions, breaking the iptables-nft translation path that previously
  succeeded.
- `iptables` 1.8.12 (shipped in Talos 1.13) and `iptables-nft` 1.8.8 (in the
  Cilium 1.18.6 / 1.19.3 image) interact slightly differently across versions.
- Talos 1.13 has a "fix: correctly calculate end ranges for nftables sets"
  commit, suggesting Talos itself uses raw nftables sets in places that may
  have changed shape with that fix.

We did not pin down which of these specifically flipped the bit. What is
established is that Cilium 1.18.6 and 1.19.3 both fail the same way on the
post-upgrade nat table, so the trigger is something in Talos 1.13's stack, not
in the Cilium binary.

### Step 6 — confirm Flux liveness is a separate problem

```bash
kubectl -n flux-system get networkpolicy
# allow-egress    <none>                                   122d
# allow-scraping  <none>                                   122d
# allow-webhooks  app=notification-controller              122d

kubectl -n flux-system get networkpolicy allow-egress -o yaml
```

`allow-egress`:

```yaml
spec:
  podSelector: {}                  # selects every pod in the namespace
  policyTypes: [Ingress, Egress]
  ingress:
    - from:
        - podSelector: {}          # only intra-namespace pods
  egress:
    - {}                           # all egress
```

With *any* policy selecting the pod, all unmatched ingress is denied. Kubelet
liveness probes come from the host network (`10.244.0.1` or the node IP),
which matches **none** of these `from` clauses. Cilium 1.19's NetworkPolicy
enforcement is strict enough to honor this — the probes time out.

You can verify this exactly via:

```bash
kubectl -n flux-system get events --sort-by='.lastTimestamp' \
  | grep -i 'unhealthy\|probe\|liveness'
# Warning   Unhealthy  ...  Readiness probe failed: Get "http://10.244.0.X:9440/readyz":
#                          context deadline exceeded
```

The fix is removing the policies (PR #204 / #205). The other Flux NetworkPolicy
config the bootstrap manifest leaves in place is harmless.

## Resolution

1. **PR #204** — `installIptablesRules: false` on Cilium HelmRelease (turned out
   to be a no-op; reverted by #208). Real fix: `bpf.masquerade=true` and
   removal of two NetworkPolicies (`allow-egress`, `allow-scraping`).
2. **PR #205** — also remove `allow-webhooks` NetworkPolicy. Without this,
   `notification-controller` stayed in CrashLoopBackOff because its `from:
   namespaceSelector: {}` does not match the host network either.
3. **PR #208** — revert `installIptablesRules: false` from PR #206; the helm
   chart in 1.19.3 silently drops the value.
4. **PR #211** — revert `TS_DEBUG_FIREWALL_MODE=iptables` from PR #209/#210;
   the Talos Tailscale extension image does not bundle the `iptables` binary
   so tailscaled crashes on startup with `exec: "iptables": executable file
   not found in $PATH` and Tailscale connectivity dies.

After PR #211 the cluster is in a stable, fully-Running state with one
documented cosmetic side effect.

## Why we could not silence the Cilium iptables noise

The 10-second `iptables ... (nf_tables): table 'nat' is incompatible` line
continues to appear in `kubectl -n kube-system logs ds/cilium`. This is
**cosmetic only**:

- Pod-to-external SNAT runs in eBPF via `bpf.masquerade=true` and is unaffected.
- Service routing runs in eBPF via `kubeProxyReplacement=true`.
- Cilium's iptables manager probes the nat table for cleanup of legacy rules;
  the read failure does not block any rule installation that matters.

Three approaches were tried and ruled out:

1. **Cilium chart `installIptablesRules: false`** (PR #206, reverted #208).
   The value does not exist in the 1.19.3 chart values schema. `helm get
   values` shows it is being passed, but the rendered DaemonSet and
   `cilium-config` ConfigMap never reference an `install-iptables-rules` key.
   Verified by waiting through a full Flux reconcile and confirming the error
   line continued at the same cadence.

2. **Tailscale `TS_NETFILTER_MODE=iptables`** (PR #209). Nothing happened at
   all — `tailscaled` doesn't read that variable name. The correct env var
   is `TS_DEBUG_FIREWALL_MODE`, defined in upstream
   [util/linuxfw/detector.go](https://github.com/tailscale/tailscale/blob/main/util/linuxfw/detector.go):

   ```go
   mode := envknob.String("TS_DEBUG_FIREWALL_MODE")
   // accepts "auto" | "iptables" | "nftables"
   ```

   The "Debug" prefix is historical; the env var is the supported public
   knob, used in Tailscale's own Kubernetes operator/sidecar/subnet manifests
   under `docs/k8s/*.yaml`.

3. **Tailscale `TS_DEBUG_FIREWALL_MODE=iptables`** (PR #210, reverted #211).
   Tailscale crashes on startup:

   ```
   getLocalBackend error: createEngine: router.Up: setting netfilter mode:
   could not create new netfilter:
   exec: "iptables": executable file not found in $PATH
   ```

   The `siderolabs/tailscale` extension image ships `tailscaled` and its Go
   dependencies but does **not** include the standalone `iptables` binary.
   `tailscaled` in `auto` and `nftables` modes manipulates nftables in-process
   via libnftnl and never invokes `iptables`; in `iptables` mode it execs the
   binary directly, and that exec fails fatally.

So the noise cannot be silenced from either side without external changes:

- Cilium upstream tracking issue
  [cilium/cilium#12879](https://github.com/cilium/cilium/issues/12879)
  ("Remove iptables / netfilter dependency") is open since 2020-08-13 with no
  milestone and several unchecked sub-tasks. Cilium 1.20.0-pre.1 release
  notes do not mention iptables manager changes. Not promising in the near
  term.
- Talos extension image lacks `iptables`. The would-be fix is a feature
  request to `siderolabs/extensions` to bundle iptables in the Tailscale
  extension image.

## Lessons learned

1. **Decompose multi-component outages before fixing**. The reflex narrative
   was "Talos upgrade broke everything." The more useful narrative was
   "three independent issues with three different triggers, two of which only
   become visible during the same window." Item-2 (Flux NetworkPolicy
   enforcement change) had nothing to do with the Talos OS upgrade — it was
   triggered by the Cilium upgrade. Conflating them sends investigation in the
   wrong direction.

2. **`talosctl ... post check passed` is necessary but not sufficient**.
   That message means the new Talos image booted and the kubelet liveness
   gate was satisfied. It does **not** mean every pod recovered. Always sweep
   `kubectl get pods -A | awk '!/Running|Completed/'` before walking away from
   a Talos OS upgrade. The post-upgrade verification section in
   `docs/talos-operations.md` records this.

3. **Helm value name silently being dropped is a real failure mode**.
   `helm get values cilium` reflects what was passed to the chart, *not*
   what the chart actually consumed. If a value sounds like it should work
   and the symptom does not change, verify by grepping the rendered
   DaemonSet/ConfigMap for the corresponding lower-case key, not by trusting
   the values output.

4. **Read the kernel state directly when CNI logs disagree with intuition**.
   The `nft list ruleset` output via a privileged debug Job is the
   authoritative answer. Without that step we would have stopped at "Cilium
   has a bug" and missed that Tailscale is the writer.

5. **Imperative recovery is acceptable when GitOps deadlocks**. Flux could
   not roll out the Cilium upgrade because Flux itself was broken by the
   problem the upgrade would fix. Manual `helm upgrade` is the correct
   break-glass; the work to commit that imperative state back to git
   (`bpf.masquerade=true` in `helmrelease.yaml`, NetworkPolicies removed
   from `gotk-components.yaml`) is the second half of the same fix.

6. **Empirical verification beats reading documentation about what an env
   var does**. `TS_NETFILTER_MODE` *sounds* right and matches the Tailscale
   CLI flag name, but `tailscaled` does not read it. We did not catch this
   from documentation; we caught it because the cluster's behavior did not
   change after the apply. The fix was upstream source-grep, not
   documentation-grep.

## Open follow-ups

- **Upstream feature request**: ask `siderolabs/extensions` to bundle
  `iptables` in the Tailscale extension image so `TS_DEBUG_FIREWALL_MODE=iptables`
  becomes a viable Cilium-noise mitigation on Talos.
- **Architecture option**: drop the Talos Tailscale extension entirely and
  rely solely on the in-cluster Tailscale operator. The host loses tailnet
  access (no `talosctl` / kubectl over the 100.x address), but the
  `ts-postrouting` chain disappears from the host nat table and the Cilium
  noise stops at the source.
- **Watch upstream Cilium**: revisit
  [cilium/cilium#12879](https://github.com/cilium/cilium/issues/12879) when
  Cilium 1.20 ships, in case the iptables manager finally becomes optional.
